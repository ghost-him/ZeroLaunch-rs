# 审查报告模板

> 本文件规定 code-review skill 最终报告的结构与输出风格。主 agent 在完成 4.1 冲突检测后，按此模板生成报告。

## 设计原则

- **按决策优先级组织**：阻塞 → 建议修复 → 低优 → 信息项，让 reviewer 一次看清能不能合
- **阻塞项内按关注点分组**：逻辑正确性 / 回归 / 架构合规 / 规则违反，不同角色快速定位
- **元信息紧跟被解释的内容**：冲突复核挂在对应问题条目下，不跨节
- **大 commit 下钻的新发现同步进前置章节**：下钻章节只做补充说明，不产生"漏在后面的新问题"

---

## 报告结构

按以下顺序输出。各节内子分组无内容时写"无"；整节无内容时写"无"。

### 一、总览

以表格呈现决策层信息：

| 项目 | 内容 |
|------|------|
| **结论** | `通过` / `建议修改后合并` / `阻塞合并` |
| **变更概要** | 一句话 + 3-5 条要点说明本次变更做了什么（聚合视角；跨 commit 时按主要 commit 或子系统概述） |
| **阻塞** | N 项 |
| **高** | N 项 |
| **中** | N 项 |
| **低** | N 项 |
| **疑点** | N 项 |
| **既有问题** | N 项 |

> 若阻塞合并，在表格后加一行注明阻塞的核心原因（1 句话）。

---

### 二、阻塞项

> 必须解决才能合并。按关注点分以下子分组，各组内按严重程度降序排列。无内容的子分组整段省略。

#### 逻辑正确性

- **[阻塞]** `文件:行` — 一句话摘要
  - 证据：控制流 / 数据流 / 调用链 / 脚本输出依据
  - 影响：用户可见行为 / 运行后果
  - 建议：最小修复方向
  - 来源：Agent 2
  - [冲突复核：Agent X 认为 Y，复核后确认 Z]（如有冲突则追加此行）

#### 回归

每条追加"变更前 / 变更后 / 证明"三项（证明为何是本次引入）：

- **[阻塞]** `文件:行` — 一句话摘要
  - 变更前：旧行为 / 旧契约
  - 变更后：新行为 / 新契约
  - 证明：为什么能确认是本次引入
  - 影响：用户可见后果
  - 建议：最小修复方向
  - 来源：Agent 3
  - [冲突复核：...]（如有）

#### 架构合规

每条标注违反的原则编号（P1-P6）。在"建议"中指明应回归到哪条既有抽象或原则（详见 `architecture-principles.md`）。

子分组按原则聚集：

- **P3 编译期层级**（确定性脚本结论优先）
- **[阻塞]** `[P3]` `文件:行` — 一句话摘要
  - 证据：违反了哪条原则的具体条款
  - 影响：为何阻碍演进
  - 建议：最小修复方向
  - 来源：Agent 4a
  - [冲突复核：...]（如有）

- **P2 类型边界泄漏**（确定性脚本结论优先）
- **[阻塞]** `[P2]` `文件:行` — ...

- **P4 运行时职责域直连**
- **[阻塞]** `[P4]` `文件:行` — ...
  - 来源：Agent 4b

- 其余原则（P1/P5/P6）的阻塞级发现按相同格式列于本子分组内。

#### 规则违反（A 类）

> A 类 = 代码违反规则，需改代码。来源：Agent 5。

- **[阻塞]** `[A 类]` `文件:行` — 一句话摘要
  - 规则原文：引用规则的具体条款
  - 代码现状：代码实际做了什么
  - 判定依据：为什么判定为 A 类
  - 建议：最小修复方向
  - 来源：Agent 5
  - [冲突复核：...]（如有）

---

### 三、建议修复

> 高/中优先级。同样按关注点分以下子分组，各组内按严重程度降序。无内容的子分组整段省略。

#### 高优先级

子分组：逻辑 / 回归 / 架构合规 / 规则违反(A 类)。格式同上"阻塞项"各子分组，仅严重程度改为 `[高]`。

#### 中优先级

子分组同上，严重程度改为 `[中]`。

---

### 四、低优先级

子分组：逻辑 / 回归 / 架构合规 / 规则违反(A 类)。格式同上，严重程度为 `[低]`。

---

### 五、疑点与既有问题

> 信息性章节，不影响合并决策，但需 reviewer 知情。

#### 疑点但证据不足

按维度（逻辑 / 回归 / 架构 / 规则）分组列出，每条保留维度标注：

- **[逻辑]** 疑点描述 | 还缺什么证据
- **[架构]** 疑点描述 | 还缺什么证据
- **[规则]** 疑点描述 | 还缺什么证据

无则写"无"。

#### 既有问题（非本次引入）

按维度（逻辑 / 回归 / 架构 / 规则）分组列出：

- **[逻辑]** 问题 | 为何判断为既有
- **[架构]** 问题 | 对应原则 | 为何判断为既有
- **[规则]** 问题 | 对应规则文件 | 为何判断为既有

无则写"无"。

---

### 六、规则更新建议（B 类）

> B 类 = 规则已过时，需更新规则。来源：Agent 5。

每条：规则文件 | 当前规则描述 | 偏离现状 | 应更新方向。

无则写"无"（若 B 类也无则整节"无"）。

---

### 七、大 commit 下钻（如有，无则省略整节）

> 仅在多 commit 范围审查且存在大 commit 时输出。

每条格式：

```
### commit <sha> — <subject>
- 结论：通过 / 建议修改 / 阻塞
- 与聚合审查关系：问题源头 / 已在聚合审查覆盖 / 新增细节
- 关键发现摘要
```

**重要**：若下钻发现聚合审查未覆盖的新增细节，这些发现必须同步出现在前面**阻塞项 / 建议修复 / 低优先级**对应分组中，并在条目末尾标注 `[下钻]`（如 `来源：大 commit 下钻`）。下钻本节只做补充说明，不在此处首次曝光需修复的问题。

---

### 八、建议手动验证清单

3-8 条最值得回归测试的行为路径，编号列出。每条标注关联的问题位置，格式：

```
1. <行为路径描述> — 关联：[阻塞·逻辑] `文件:行`、[高·架构] `文件:行`
2. <行为路径描述> — 关联：[高·回归] `文件:行`
...
```

---

## 输出风格

- 先给结论，再给证据
- 建设性、可执行；优先说明"为什么是问题"和"最小修复方向"
- 不把代码风格细节包装成架构问题
- 没有证据就放进"疑点但证据不足"，不用"可能有问题"替代
- 对"回归"一词保持严格：**在正式输出前，必须在思考中证明是本次引入**
- 每条问题在报告中只出现一次，不跨节重复
- 空节写"无"；各节间用 `---` 分隔，保持视觉层次清晰
- 冲突复核直接挂在对应问题条目下（追加一行 `[冲突复核：...]`），**不跨节**
- 大 commit 下钻新增的发现必须同步进前置章节

---

## 报告文件头部元数据

写入文件时，文件开头加一行：

```
<!-- 审查日期: YYYY-MM-DD | 范围: <范围> | 结论: <通过/建议修改/阻塞> -->
```
