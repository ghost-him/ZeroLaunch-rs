# AGENTS.md - AI 协作指南

本文档旨在引导 AI 助手快速理解 **ZeroLaunch-rs** 项目架构，并明确协作中的工程规范。

## 一、 项目概览

**ZeroLaunch-rs** 是基于 Tauri + Vue 3 的 Windows 快捷启动器。

- **后端 (Rust/Tauri)**: `src-tauri/` - 核心逻辑、插件系统、数据检索。
- **前端 (Vue 3/TS)**: `src-ui/` - 用户界面与交互。
- **核心能力**: 多源搜索（UWP/文件/书签）、语义搜索（AI）、Everything 集成、多端同步。

## 二、 插件架构 (Plugin System)

项目采用基于 Trait 的可插拔架构，主要组件定义在 `src-tauri/src/plugin_system/types.rs`：

| Trait              | 职责                                                           |
| :----------------- | :------------------------------------------------------------- |
| `Configurable`     | **基础**: 统一配置管理（`setting_schema`, `apply_settings`）。 |
| `DataSource`       | **提供者**: 产出原始搜索候选项（Candidate）。                  |
| `KeywordOptimizer` | **处理器**: 对查询关键词进行清洗或扩展（如去版本号、拼音化）。 |
| `SearchEngine`     | **匹配器**: 计算 Candidate 与关键词的匹配得分。                |
| `ScoreBooster`     | **排序器**: 基于用户习惯进行二次分值提升。                     |
| `ActionExecutor`   | **执行器**: 定义如何执行动作（如启动程序、唤醒窗口）。         |
| `Plugin`           | **独立功能**: 处理特定指令的完整闭环插件。                     |

## 三、 协作准则 (核心约束)

### 3.1 行为优先级
1. **深度理解先行 (Deep-Read)**：在第一轮阅读或执行任务前，**必须深度研读 `README.md` 及项目整体目录结构**，确保对上下文有全局认知。
2. **最小化改动原则**：代码更改应保持克制，优先在现有框架内解决问题。除非必要或用户明确要求，否则避免大面积重构。
3. **设计确认流程**：
   - 接收任务 -> 分析代码 -> 输出设计方案/修改计划 -> **等待用户确认** -> 执行更改。
4. **任务总结**：任务完成后，须总结目标与达成情况，并提供符合规范的 Commit 信息，但不可代替用户完成 Commit 操作。
5. **文档同步原则**：若所做的改动涉及现有功能逻辑、配置项或接口的变更，且项目内存在对应的介绍文档（如 README.md 或插件说明），必须同步修正相关文档内容，确保代码实现与文档描述的绝对一致。

### 3.2 软件工程规范
1. **工程化命名**：命名必须符合软件工程范式。变量、函数及类名需具备**语义化、一致性**，并遵循对应技术栈（Rust 蛇形命名 / TS 驼峰命名）的业界标准。
2. **配置处理安全**：读取 JSON 数值配置时，统一使用 `as_f64()` 而非 `as_i64()`（预防前端浮点数传入导致的解析失败）。
3. **注释规范**：
   - 函数内部：非必要不添加注释。
   - 函数头部：**必须**包含简要功能描述、参数及返回值含义。

## 四、 语言与交付格式

- **交流语言**：全程使用中文。
- **界面文本/注释**：面向用户的 UI 字符串使用中文。
- **Commit 格式**：使用 `<type>: <subject>` 格式（中文描述）。
  - `feat`: 新功能 | `fix`: 修复 | `refactor`: 重构 | `docs`: 文档 | `style`: 格式 | `chore`: 事务。
