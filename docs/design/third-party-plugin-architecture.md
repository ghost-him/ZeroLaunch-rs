# 第三方插件架构设想

> 状态：讨论记录，待审核修正。本文档不涉及代码实现，仅记录对第三方插件系统的组织架构设想。

---

## 1. 核心理念

第三方插件不是"在后端注册几个函数"——它必须是一个**自包含的前端+后端联合体**：

```
第三方插件 = 前端部分（Vue 组件） + 后端部分（Plugin trait 实现）
```

插件作者同时交付 UI 和逻辑，宿主（ZeroLaunch）提供平台能力接口和生命周期管理。

---

## 2. 组织架构

### 2.1 三层模型

```
┌──────────────────────────────────────────────┐
│  宿主（ZeroLaunch Core）                      │
│  - 窗口管理（显示/隐藏/定位）                   │
│  - 会话生命周期（搜索栏唤醒/重置）              │
│  - 平台能力注入（HostApi → PluginHandle）      │
│  - 插件注册与发现                             │
└──────────────┬───────────────────────────────┘
               │ Plugin trait + PluginHandle
┌──────────────▼───────────────────────────────┐
│  插件后端（Plugin Backend）                    │
│  - 实现 Plugin trait（query / execute_action）│
│  - 通过 PluginHandle 调用平台能力              │
│  - 定义触发关键词与元数据                       │
│  - 返回 QueryResponse 给宿主                   │
└──────────────┬───────────────────────────────┘
               │ QueryResponse（含前端渲染指令）
┌──────────────▼───────────────────────────────┐
│  插件前端（Plugin Frontend）                   │
│  - Vue 3 组件（由宿主动态加载）                │
│  - 自行管理所有键盘事件（不需要宿主拦截按键）     │
│  - 通过 bridge 命令与自身后端通信               │
│  - 决定自身的显示模式（行内/全页面）             │
└──────────────────────────────────────────────┘
```

### 2.2 宿主的职责边界

宿主只负责 **框架性事务**，不介入插件的内部交互：

| 职责 | 说明 |
|------|------|
| 插件发现与加载 | 扫描插件目录，加载插件清单，实例化前后端 |
| 窗口生命周期 | 显示/隐藏搜索窗口，窗口定位与大小 |
| 搜索栏管理 | 提供搜索输入框，将原始查询文本传给后端路由 |
| 触发词路由 | 匹配触发关键词 → 激活对应插件的 query |
| 平台能力注入 | 通过 `PluginHandle` 向后端提供文件、Shell、图标等能力 |
| 结果区域挂载 | 根据插件返回的模式（行内/全页面）挂载插件组件到对应 DOM 区域 |

宿主**不负责**：
- 插件内部按键处理（方向键、快捷键、Enter/Escape 等）
- 插件 UI 的具体渲染逻辑
- 插件的自定义状态管理

### 2.3 插件的职责边界

| 职责 | 说明 |
|------|------|
| 后端 query | 接收搜索词，返回 `QueryResponse`（List / CustomPanel / Empty） |
| 后端 execute_action | 执行具体操作（复制结果、打开文件等） |
| 后端配置 | 通过 `Configurable` trait 暴露可配置项 |
| 前端组件 | 自包含 Vue 组件，处理自身所有交互 |
| 前端键盘管理 | 组件内部通过标准 Vue 事件绑定处理所有按键 |
| 前端-后端通信 | 通过已有的 bridge IPC 命令与后端交互 |

## 3. 两种显示模式

插件通过 `QueryResponse` 声明自己的显示模式：

### 3.1 行内模式（Inline Plugin）

```
┌─────────────────────────────────┐
│ 搜索栏：[= 1+2        ]         │  ← 宿主提供搜索栏
├─────────────────────────────────┤
│ 插件结果面板（嵌入在搜索栏下方）   │  ← 插件组件挂载点
│ ┌─────────────────────────────┐ │
│ │ 3                           │ │  ← 插件自行渲染
│ │ [复制结果]                   │ │
│ └─────────────────────────────┘ │
│                                 │
│ （下方正常显示其他搜索结果）       │  ← 可选：继续显示搜索候选项
└─────────────────────────────────┘
```

- 对应 `keep_search_bar = true`（`CustomPanel`）
- 搜索栏保留并继续可用
- 插件组件挂在结果列表区域的上方或替代结果列表
- 适用场景：计算器、翻译、单位换算等"输入-结果"型插件

### 3.2 全页面模式（Full Page Plugin）

```
┌─────────────────────────────────┐
│                                 │
│                                 │
│    插件接管整个窗口              │
│    自行渲染所有 UI               │
│    自行管理所有按键              │
│                                 │
│           [退出]                │  ← 插件自行提供退出按钮/快捷键
│                                 │
└─────────────────────────────────┘
```

- 对应 `keep_search_bar = false`（`CustomPanel` 或未来的变体）
- 隐藏搜索栏，窗口全部区域交给插件组件
- 插件组件通过标准 Vue 事件处理所有键盘输入
- 宿主不再拦截任何按键（包括 Escape）
- 退出方式：插件调用 `store.exitFullPagePlugin()` 或宿主提供统一的退出入口
- 适用场景：笔记、待办事项、文件浏览器等需要完整 UI 的插件

## 4. 按键处理策略

### 4.1 当前原则

- **搜索模式 + 行内参数模式 + 参数面板模式**：宿主通过 `useKeyboardRouter` 统一管理
- **行内插件模式**：宿主保留搜索栏，Enter/Escape 由宿主拦截（因为搜索栏需要响应），插件内部只处理结果区域的交互
- **全页面插件模式**：宿主**完全不拦截**任何按键。所有键盘事件由插件的 Vue 组件通过标准 `@keydown` 绑定处理

### 4.2 全页面模式的退出

插件自行决定退出方式，宿主不硬编码 Escape。常见的退出模式：
- 插件在 UI 上提供关闭按钮
- 插件在组件内绑定 Escape → 调用 `store.exitFullPagePlugin()`
- 插件在组件内绑定自定义快捷键退出

宿主只在以下情况兜底：
- 窗口失焦 → 自动隐藏（由宿主窗口管理处理，不涉及按键事件）
- 全局热键再次触发 → 切换窗口可见性

### 4.3 为什么不需要后端的 bridge_plugin_key_event

插件的前端组件本身就活在浏览器的 DOM 环境中，键盘事件自然会被插件组件捕获。把按键通过 IPC 送到后端、后端再传给插件的后端、插件后端再返回响应给前端——这条路是绕远路。

正确的链路：
```
用户按键 → DOM 事件冒泡 → 插件 Vue 组件 @keydown 处理 → 返回 void（已处理）
```

而不是：
```
用户按键 → 前端全局拦截 → IPC → 后端路由 → 插件后端 → IPC 返回 → 前端处理响应
```

唯一的例外是**行内插件模式**的 Enter/Escape——这些键需要和搜索栏共享，所以由宿主统一拦截是合理的。但这是前端内部的路由决策，不需要经过后端。

## 5. 插件加载机制设想

### 5.1 插件包结构

```
plugins/
└── my-plugin/
    ├── manifest.json        ← 元数据（id, name, version, trigger_keywords 等）
    ├── backend.js           ← 插件后端（或 .wasm / 动态库，取决于技术选型）
    └── frontend/
        ├── index.js         ← Vue 组件入口
        └── styles.css       ← 插件样式
```

### 5.2 热加载

- 插件目录监控 → 新增/更新/删除 → 自动重载
- 重载流程：停止旧插件实例 → 释放资源 → 重新初始化

### 5.3 隔离性

- 每个插件运行在独立上下文（iframe / Shadow DOM / Web Worker，待定）
- 插件之间不能直接通信，只能通过宿主提供的事件总线
- 插件崩溃不影响宿主和其他插件

## 6. 插件与宿主的通信接口

### 6.1 后端接口（Plugin trait）

当前已定义的核心方法（`plugin_system/types.rs`）：

- `metadata()` — 返回插件元数据（id, name, version, trigger_keywords 等）
- `init(ctx, host_api)` — 初始化，获取平台能力句柄
- `query(ctx, query)` — 处理搜索查询，返回 `QueryResponse`
- `execute_action(ctx, action_id, payload)` — 执行动作

未来可能的扩展：
- `on_mode_enter()` / `on_mode_exit()` — 模式切换生命周期回调
- `get_frontend_entry()` — 返回前端组件入口路径（用于动态加载）

### 6.2 前端接口

插件前端组件需要遵循的约定：
- 导出一个 Vue 3 组件作为默认导出
- 接收 props：`pluginData`（后端 `QueryResponse` 中的数据）、`hostActions`（宿主注入的能力）
- 通过 emit 或 store 与宿主交互（退出模式、更新面板数据等）

## 7. 当前状态与待决问题

### 7.1 当前实现

- 唯一的内置插件：`calculator_plugin`（计算器）
- 运行在行内模式（`keep_search_bar = true`）
- 插件前端为宿主内置的 `CalculatorPanel.vue`，通过 `panelType` 字段匹配
- 尚未支持第三方插件动态加载

### 7.2 待决问题

1. **插件前端加载方式**：动态 import()、iframe 隔离、还是编译时打包？
2. **插件后端运行时**：JavaScript（Bun/QuickJS）、WebAssembly、还是原生动态库？
3. **插件安全模型**：权限声明机制（类似浏览器 extension manifest）、网络访问控制、文件系统访问范围
4. **插件分发**：插件市场、手动安装、GitHub Release 自动更新
5. **版本兼容**：插件 API 版本号与宿主版本的关系，如何声明兼容性
6. **调试工具**：插件开发者如何调试（devtools 支持、日志查看、热重载）
7. **全页面插件的前端组件入口**：如何从后端 `QueryResponse` 映射到前端组件。当前通过 `panelType` 字符串匹配（如 `"calculator"`），未来第三方插件需要更通用的路由机制
